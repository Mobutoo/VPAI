### /content command:
```
### /content <format> <brief>

Cree un nouveau contenu et lance l'etape 1 (brief creatif).

**Formats valides:** reel-motion-text, reel-meme-skit, reel-feature-showcase, reel-teaser

**Workflow:**
1. Valider le format (doit etre dans la liste)
2. Appeler cf-create-content via n8n (cree ligne NocoDB + shot Kitsu)
3. Consulter Qdrant brand-voice pour le ton de la marque
4. Generer le brief structure (objectif, cible, message cle, contraintes, KPI)
5. Appeler cf-update-content pour sauvegarder le brief
6. Appeler cf-kitsu-sync pour mettre la tache Brief en "wip"
7. Presenter le brief a l'utilisateur dans Telegram
8. Attendre /ok pour valider ou /adjust pour modifier

**Reponse Telegram:**
🎬 **Nouveau contenu: <title>**
Format: <format>
Brief:
- Objectif: ...
- Cible: ...
- Message cle: ...
- Contraintes: ...

Tape /ok pour valider le brief, /adjust pour modifier.
```

### /ok command:
```
### /ok

Valide l'etape en cours et avance a la suivante.

**Workflow:**
1. Lire le contenu actuel via cf-read-content
2. Verifier que l'etape en cours a un livrable (pas vide)
3. Appeler cf-update-content: current_step += 1
4. Appeler cf-kitsu-sync: tache etape N → "done", tache etape N+1 → "wip"
5. Si l'etape suivante est automatisable (recherche, moodboard), la lancer immediatement
6. Sinon, presenter les instructions pour l'etape suivante

**Etapes auto-enchainables (l'agent les execute sans attendre):**
- Etape 2 (Recherche): search-content skill + web search
- Etape 3 (Moodboard): generate-visual skill (4-8 images de reference)
- Etape 9 (Generation assets): delegation a n8n generate-assets workflow
- Etape 10 (Rough cut): delegation a n8n rough-cut workflow

**Reponse Telegram:**
✅ Etape <N> (<nom>) validee.
Passage a l'etape <N+1> (<nom>).
[instructions specifiques a l'etape]
```

### /adjust command:
```
### /adjust <instruction>

Modifie l'etape en cours selon l'instruction donnee.

**Workflow:**
1. Lire le contenu actuel via cf-read-content
2. Appliquer la modification demandee (reecrire le livrable de l'etape)
3. Incrementer la version dans NocoDB (si scenes: scene.version += 1)
4. Appeler cf-kitsu-sync: tache → "retake" puis "wip"
5. Presenter la nouvelle version

**Reponse Telegram:**
🔄 Etape <N> (<nom>) modifiee (v<version>).
[nouveau contenu]
Tape /ok pour valider, /adjust pour re-modifier.
```

### /back command:
```
### /back <step_number>

Retourne a une etape anterieure avec analyse d'impact.

**Workflow:**
1. Lire le contenu actuel
2. Calculer l'impact via la matrice d'invalidation
3. Presenter l'impact AVANT d'agir (lecture seule)
4. Attendre confirmation /go ou /cancel

**Si cross-phase (retour avant une gate passee):**
- Necessite un unlock explicite
- Montrer l'impact complet

**Reponse Telegram:**
⚠️ Retour a l'etape <N> (<nom>).
Impact:
- Etapes invalidees: <liste>
- Scenes a refaire: <liste>
- Scenes conservees: <liste>

Tape /go pour confirmer, /cancel pour annuler.
```

```
### /impact

Affiche ce qui serait invalide si l'etape en cours etait modifiee (lecture seule, ne modifie rien).

**Workflow:**
1. Lire le contenu actuel
2. Calculer la cascade d'invalidation
3. Presenter sans agir

**Reponse Telegram:**
🔍 Impact si modification de l'etape <N> (<nom>):
```

### Gate commands:
```
### Gates

#### /lock-preprod
Verrouille les etapes 1-5 (Pre-production).
**Pre-requis:** etapes 1-5 toutes en statut "done".
**Effet:** NocoDB phases 1-5 → "locked", Kitsu taches 1-5 → "locked".

#### /lock-script
Verrouille les etapes 6-8 (Ecriture).
**Pre-requis:** gate preprod verrouillee + etapes 6-8 en "done".
**Effet:** NocoDB phases 6-8 → "locked", nombre de scenes fige.

#### /ok-rough
Valide le rough cut (etape 10).
**Pre-requis:** gate script verrouillee + etape 10 en "done".
**Effet:** structure du montage figee, ajustements restants = cosmetiques.

#### /ok-final
Valide le montage final (etape 12 review).
**Pre-requis:** gate rough verrouillee + etape 12 en "done".
**Effet:** export haute qualite lance.

#### /published
Marque le contenu comme publie.
**Pre-requis:** gate final validee + etape 14 en "done".
**Effet:** NocoDB status → "published", Kitsu projet archive, embedding script → Qdrant.

**Reponse gate validee:**
🔒 Gate <nom> verrouillee.
Etapes <N>-<M> en lecture seule.
Prochaine etape: <N+1> (<nom>).

**Reponse gate refusee:**
❌ Impossible de verrouiller <nom>.
Etapes manquantes: <liste etapes pas en "done">.
```

```
## Matrice d'Invalidation

| Modification de... | Ce qui est invalide |
|--------------------|---------------------|

| Recherche (2) | Moodboard (3) → Concept (4) → Script (6) → Storyboard (7) → Assets (9) → Montage (10-11) |
| Moodboard (3) | Concept (4) → Script (6) → Storyboard (7) → Assets (9) → Montage (10-11) |

| Casting (5) | Script dialogues (6) → Storyboard (7) → Assets scenes avec persos (9) |
| Script (6) | Storyboard (7) → Assets (9) → Montage (10-11) |
| 1 scene storyboard (7) | Uniquement l'asset + portion montage de cette scene |
| Sound design (8) | Musique/SFX dans le montage uniquement (10-11) |
| 1 asset (9) | Portion montage de cette scene uniquement |

**Regle:** ne jamais invalider plus que necessaire. Si une scene n'est pas impactee, elle est preservee.
```

### Rules section:
```
## Regles

1. **Delegation obligatoire**: toutes les operations NocoDB et Kitsu passent par n8n (secrets pas dans le sandbox)

9. **Cross-phase unlock**: retourner avant une gate verrouillée necessite un unlock explicite avec analyse d'impact complete
10. **Archivage Qdrant**: a la publication, le script valide est embedde dans brand-voice pour la memoire creative
```
