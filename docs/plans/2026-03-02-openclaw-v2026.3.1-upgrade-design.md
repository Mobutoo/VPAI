# Design : Montée en version OpenClaw v2026.2.23 → v2026.3.1

**Date** : 2026-03-02
**Auteur** : Claude Opus 4.6 (brainstorming session)
**Approche** : In-place atomique (Approche A)
**Scope** : Phase 1 — Upgrade uniquement. Phase 2 (intégration Plane workspace Ewutelo) séparée.

## Contexte

- **Version actuelle** : `ghcr.io/openclaw/openclaw:2026.2.23`
- **Version cible** : `ghcr.io/openclaw/openclaw:2026.3.1` (released 2026-03-02)
- **Infrastructure** : OVH VPS 8GB, Docker Compose, Caddy reverse proxy, VPN Tailscale
- **REX existants** : 10+ sessions documentées (REX-45 à REX-53, sections 11.1-11.23 TROUBLESHOOTING.md)

## Décisions architecturales

### Phasage en 2 temps

| Phase | Scope | Session |
|-------|-------|---------|
| **Phase 1** (ce plan) | Upgrade OpenClaw + heartbeat infra-only + healthcheck HTTP + Telegram DM topics + sandbox fixes | Actuelle |
| **Phase 2** (à venir) | Intégration Plane workspace **Ewutelo** — agents OpenClaw créent/gèrent des tâches | Séparée |

**Raison** : Le heartbeat natif (v2026.3.1) appelle HEARTBEAT.md qui référence Kaneo API (morte). Combiner upgrade + Plane = couplage de risques non diagnosticable. 2 phases isolent chaque cause de failure.

### Workspaces Plane — distinction critique

- **Javisi** = workspace Plane existant pour une autre application. **NE PAS TOUCHER.**
- **Ewutelo** = workspace Plane pour les agents OpenClaw (Phase 2). URL : `work.ewutelo.cloud`
- Concierge (Mobutoo) = admin du workspace Ewutelo
- Autres agents = membres du workspace Ewutelo

### 6. HEARTBEAT.md.j2 — Réécriture infra-only

Version infra-only qui ne fait AUCUN appel à Kaneo ou Plane. Le concierge vérifie uniquement la santé infrastructure.

```markdown
# Heartbeat — Infrastructure Health Check

Tu es le concierge de la stack {{ project_name }}. Ceci est un heartbeat automatique.

## Vérifications obligatoires (dans l'ordre)

### 1. Santé des containers Docker
Vérifie que tous les services critiques sont "healthy" :
- `docker ps --filter health=unhealthy --format "table {% raw %}{{.Names}}\t{{.Status}}{% endraw %}"`
- Services critiques : postgresql, redis, qdrant, litellm, n8n, caddy, openclaw
- Si un service est unhealthy : note-le et vérifie ses logs (`docker logs <name> --tail 10`)

### 2. Espace disque
- `df -h /opt/{{ project_name }}`
- ALERTE si usage > 85%
- CRITIQUE si usage > 95%

### 3. Mémoire disponible
- `free -m | grep Mem`
- ALERTE si available < 512MB
- CRITIQUE si available < 256MB

### 4. Budget IA (si n8n accessible)
- Délègue à n8n via le skill budget-monitor
- Vérifie le spend 24h via LiteLLM `/global/spend`

## Format de rapport

Résume en 3 lignes maximum :
- Ligne 1 : état global (✅ OK / ⚠️ WARNING / 🔴 CRITICAL)
- Ligne 2 : services unhealthy (si aucun : skip)
- Ligne 3 : métriques clés (disk, RAM, budget 24h)

## Restrictions
- PAS d'appels API externes
- PAS de project management (intégration Plane prévue Phase 2)
- PAS de création de tâches ou tickets
- Durée max : 2 minutes
```

## Bugs résolus par cette upgrade

| Bug | REX | Résolution v2026.3.1 |
|-----|-----|----------------------|
| Heartbeat/cron non supportés | REX 11.19 | `heartbeat.lightContext` officiel |
| Docker image permissions EACCES | REX-48/49 | Extensions dirs normalisés 755/644 |
| Sandbox browser needs `--no-sandbox` | Nouveau | `OPENCLAW_BROWSER_NO_SANDBOX=1` + epoch bump |
| Agent thinking failures | Nouveau | Retry `think=off` quand provider rejette |
| Gateway WS floods | Nouveau | Close repeated post-handshake unauthorized floods |
| Auto-reply `NO_REPLY` token leak | Nouveau | Strip du token contrôle dans messages mixtes |
