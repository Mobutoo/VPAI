**cf-create-content.json** — 3 nodes: Webhook Trigger + Create Content (Code) + Webhook Response

Node ID prefix: `cfcc-`
Webhook path: `cf-create-content`

Code node logic (Create Content):
1. Validate secret (dual pattern: body.secret or header x-webhook-secret)
2. Extract `brand_id`, `title`, `format` (default "reel"), `brief` from body
3. Resolve table IDs: use `$env.NOCODB_CONTENTS_TABLE_ID` and `$env.NOCODB_CF_BASE_ID` if set, otherwise call NocoDB meta API to discover by base name "content-factory" and table name "contents"
4. POST to NocoDB records endpoint to create row with fields: `{ brand_id, title, format, status: "brief", current_step: 1, brief: JSON.stringify(brief) }`
5. Return `{ status: "ok", content_id: result.Id }` (NocoDB auto-increment Id)

Response node: respondWith "json", responseBody returns the Code node output.

**cf-read-content.json** — 4 nodes: Webhook Trigger + Validate Secret (Code) + Read Content and Brand (Code) + Webhook Response

Node ID prefix: `cfrc-`
Webhook path: `cf-read-content`

Validate Secret node: standard secret validation, extract `content_id` and `action` (default "read_content").

Read Content and Brand node (Code):
1. Resolve table IDs (same pattern as cf-create-content -- env var or meta API fallback)
2. GET NocoDB records for contents table filtered by `(Id,eq,{content_id})`, limit 1
3. If content row has `brand_id`, do a second GET to brands table filtered by `(Id,eq,{brand_id})`, limit 1
4. Merge content fields + brand fields (prefixed: `brand_tone` from `tone`, `brand_name` from `name`, `brand_palette` from `palette`, `brand_typography` from `typography`)
5. Return merged object with `content_id` set to the row Id

Response node: respondWith "json", returns the merged content+brand object.

**cf-update-content.json** — 3 nodes: Webhook Trigger + Update Content (Code) + Webhook Response

Node ID prefix: `cfuc-`
Webhook path: `cf-update-content`

Code node logic (Update Content):
1. Validate secret -- MUST support BOTH patterns:
   - Pattern A: `body.secret` (used by cf-brief-to-concept, cf-script-to-storyboard)
   - Pattern B: `headers['x-webhook-secret']` (used by cf-invalidation-engine, cf-kitsu-inbound)
2. Extract `content_id` from body (required, throw if missing)
3. Extract `fields` object from body (the key-value pairs to update)
4. If `fields` contains JSON objects, stringify them before sending to NocoDB
5. Resolve table IDs (same resolveTableIds pattern as cf-create-content)
6. PATCH to NocoDB records endpoint: `{ Id: parseInt(content_id), ...fields }`
7. Return `{ status: "ok", content_id }`

The `action` field in body is OPTIONAL (defaults to "update_content") and is used for routing only -- it must NOT be included in the PATCH fields.

**cf-scene.json** — 4 nodes: Webhook Trigger + Validate and Route (Code) + Execute Scene Action (Code) + Webhook Response

Node ID prefix: `cfsc-`
Webhook path: `cf-scene`

Validate and Route node:
1. Validate secret (dual pattern)
2. Extract routing `action` FIRST from body: one of `create_scene`, `list_scenes`, `update_scene`, `invalidate_scene`
3. Extract `content_id` from body (required for all actions)
4. Pass action + content_id + remaining body fields forward

CRITICAL Pitfall 2 from research: The `create_scene` caller sends a field ALSO called `action` which is the scene action description text. The routing action and scene description field share the same name. Resolution:
- Extract routing action as `route_action = body.action` BEFORE processing scene fields
- For `create_scene`: use `body.scene_action` for the scene action text (the caller sends `scene_action` per the corrected API contract in the research), OR if body contains `action` that is a known route action, strip it from scene fields

Execute Scene Action node — action routing in a single Code node:

**create_scene:**
- POST to NocoDB scenes table: `{ content_id, scene_number, description, duration_sec: body.duration, dialogue, screen_text: body.screen_text || '', transition, status: body.status || 'draft', provider: body.provider || '' }`
- Return `{ status: "ok", scene_id: result.Id }`

**list_scenes:**
- GET NocoDB scenes records filtered by `(content_id,eq,{content_id})`, sorted by `scene_number` ascending
- Return `{ scenes: records.list }`

**update_scene:**
- Find scene by content_id + scene_number: GET with `(content_id,eq,{content_id})~and(scene_number,eq,{scene_number})`
- PATCH with provided fields: `{ Id: sceneRow.Id, asset_url, provider, status, ... }`
- Return `{ status: "ok", scene_id: sceneRow.Id }`

**invalidate_scene:**
- If `scene_id` provided: PATCH that row directly with `{ Id: parseInt(scene_id), status: "invalidated", asset_url: "" }`
- Return `{ status: "ok", scene_id }`

Both workflow JSON files MUST follow the same n8n JSON structure, settings, tags, and node type versions as cf-create-content.json created in Task 2.
